# Card Definition Pipeline & Scaling Strategy

> How card definitions are structured, authored, and organized — and how this
> approach scales from 112 cards to 27,000+.

---

## The Rust DSL Decision

Card definitions are hand-authored Rust code using the engine's Effect DSL. This
is a deliberate choice with significant development advantages:

### What the Rust DSL gives you

1. **Compile-time correctness.** Typo an enum variant (`Effect::DealDmage`), add
   a wrong field, or pass the wrong type — `cargo check` catches it instantly.
   A JSON equivalent with `"effect": "deal_dmage"` would silently pass until runtime.

2. **IDE support.** Autocomplete on `Effect::`, `TriggerCondition::`, `Cost::` lets
   you explore what the engine supports while authoring. Data formats give you nothing.

3. **Refactoring safety.** Rename an Effect variant or add a required field to
   `CardDefinition` — the compiler shows you every affected card definition. With
   a data format, you'd need a separate schema validator and might miss things.

4. **Definitions are partial tests.** A card definition that compiles proves the
   DSL can express the card's behavior. That's free verification.

5. **Agent reliability.** Counterintuitively, an agent writing Rust that gets
   compile-checked is more reliable than one writing JSON that silently accepts
   garbage. The agent's failure mode is "doesn't write the file" — not "writes
   invalid Rust." When it does write, the type system constrains it toward
   correctness.

### What it costs

- **Compile time scales with card count.** Every card compiles into the binary,
  whether or not a given match uses it. At 112 cards this is invisible; at 27,000
  it would matter.
- **Authoring requires Rust knowledge.** A data format could be authored by
  non-Rust tools (web editors, Python scripts). The Rust DSL requires understanding
  the type system.

### Why we keep it

The engine's Effect DSL is still evolving — new variants are added with each
ability implementation. The compiler flagging every affected card definition when
the DSL changes is more valuable than any convenience a data format would provide.
The cost is manageable through crate organization (see below).

## Organization: Current → Near-Term → Long-Term

### Current State (112 cards)

All cards in one file, one function, one `vec![]` literal:

```
crates/engine/src/cards/definitions.rs  (3,118 lines)
└── pub fn all_cards() -> Vec<CardDefinition> { vec![ ... ] }
```

Organized by comment headers: Mana rocks, Lands, Removal, Counterspells, Card
draw, Ramp, Equipment, Utility creatures, Replacement-effect cards, Kicker,
Flashback, Buyback.

**Problems at scale:**
- Agent struggles to find insertion points in a 3k+ line file
- Parallel authoring sessions cause merge conflicts
- Changing one card recompiles 3,118 lines
- File will grow to 15k+ lines at 500 cards

### Near-Term Target (100–2,000 cards): Separate Crate, One File Per Card

```
crates/
├── engine/                 (rules, state, effects — changes often)
├── card-defs/              (card definitions — changes when authoring)
│   ├── Cargo.toml          (depends on engine for DSL types)
│   ├── build.rs            (scans cards/, generates mod.rs)
│   └── src/
│       ├── lib.rs          (pub fn all_cards() — aggregates all card modules)
│       ├── helpers.rs      (cid, types, creature_types, mana_pool, etc.)
│       └── cards/
│           ├── mod.rs      (auto-generated by build.rs)
│           ├── sol_ring.rs
│           ├── arcane_signet.rs
│           └── ...
```

Each card file is ~20-30 lines:

```rust
use crate::helpers::*;

pub fn card() -> CardDefinition {
    CardDefinition {
        card_id: cid("sol-ring"),
        name: "Sol Ring".to_string(),
        mana_cost: Some(ManaCost { generic: 1, ..Default::default() }),
        types: types(&[CardType::Artifact]),
        oracle_text: "{T}: Add {C}{C}.".to_string(),
        abilities: vec![AbilityDefinition::Activated {
            cost: Cost::Tap,
            effect: Effect::AddMana {
                player: PlayerTarget::Controller,
                mana: mana_pool(0, 0, 0, 0, 0, 2),
            },
            timing_restriction: None,
        }],
        ..Default::default()
    }
}
```

**Why this works:**
- Agent writes one file, never touches any other
- Zero merge conflicts between parallel sessions
- `build.rs` auto-discovers card files — no manual module declarations
- Incremental compilation: engine changes don't recompile cards, card changes
  don't recompile the engine
- Full Rust DSL preserved: type checking, IDE support, refactoring safety
- Two call sites change trivially: `cards::definitions::all_cards()` →
  `card_defs::all_cards()`

**`build.rs` behavior:** Scan `src/cards/` for `.rs` files (excluding `mod.rs`),
generate `mod.rs` with `pub mod <name>;` declarations and an `all()` function that
calls each module's `card()` and collects them into a `Vec`.

### Long-Term Option (2,000–27,000 cards): When to Worry

At the "every printable Magic card" scale (~27,000 cards), the Rust DSL approach
has real costs:

| Concern | ~500 cards | ~2,000 cards | ~10,000 cards | ~27,000 cards |
|---------|-----------|-------------|--------------|--------------|
| Compile time (incremental) | Negligible | Noticeable | Slow | Painful |
| Compile time (clean build) | ~10s | ~30s | ~2min | ~5min+ |
| Binary size | ~500KB | ~2MB | ~8MB | ~20MB+ |
| `cargo check` | Fast | Fine | Slow | Painful |
| File count | Manageable | Fine | Many | Needs subdirs |

**At 2,000+ cards**, consider:
- Subdirectories: `cards/creatures/`, `cards/instants/`, `cards/artifacts/`, etc.
- Feature flags: `card-defs = { features = ["core-500", "extended"] }` to compile
  subsets
- Parallel compilation: Rust compiles modules in parallel, so many small files
  actually compile faster than one huge file

**At 10,000+ cards**, seriously evaluate:
- Runtime loading with `serde::Deserialize` on `CardDefinition` (or an intermediate
  `CardData` struct). Cards as `.ron` or `.json` files loaded at startup. Compile
  time becomes constant. Schema validation replaces type checking.
- Keep the Rust DSL for the ~500 most complex cards (nested conditionals, complex
  replacement effects) and use data-driven loading for simple keyword creatures.

**Key architectural constraint:** `CardRegistry::new(impl IntoIterator<Item =
CardDefinition>)` doesn't care where definitions come from. Migration from Rust
source to runtime loading is a swap at the loading layer, not a rewrite of the
engine. This is the escape hatch that makes it safe to start with the Rust DSL.

## Authoring Pipeline

### Scryfall Skeleton Generation

Most of a CardDefinition is mechanical data available from Scryfall: name, types,
subtypes, mana cost, cmc, power/toughness, oracle text, keywords. Only the
`abilities` vec requires human/agent authoring.

A skeleton generator script takes a card name, fetches Scryfall data, and writes
a `.rs` file with all mechanical fields filled in and `abilities: vec![]` with
TODO comments derived from oracle text keywords.

This cuts per-card authoring effort roughly in half — the author only fills in
the behavior, not the boilerplate.

### Ability-Gated Worklist

Not every card can be authored at any given time. Cards that use unimplemented
keywords must wait. The worklist classifies cards as:

- **Ready**: All keywords are `validated` or `complete` in the engine
- **Blocked**: Has >= 1 keyword with status `none` — author after ability is
  implemented
- **Deferred**: Needs Morph, Mutate, Phasing, or Transform — explicitly deferred
  mechanics

The worklist is regenerated whenever abilities are implemented or the test deck
set changes. It drives authoring priority: ready cards sorted by deck frequency.

### Agent Authoring

The `card-definition-author` agent translates oracle text into the Effect DSL.
After the refactor to one-file-per-card, the agent's task simplifies from "find
the right insertion point in a 3k line file and splice in a definition" to "write
a complete small file." This directly addresses the ~50% failure rate observed in
batch sessions.

Performance is tracked in `_authoring_log.json` with per-run metrics: card name,
success/fail, failure type, tokens used, turns used. Batches of 5-10 cards are
run to compare agent prompt changes.

### Manual Authoring

For complex cards (nested conditionals, replacement effects, multi-step triggers),
manual authoring using existing definitions as templates is faster and more
reliable than the agent. The skeleton generator still saves time by pre-filling
boilerplate.

## Alternatives Considered

### Split by card role (Mana rocks, Removal, etc.)

Categories are subjective — is Solemn Simulacrum "ramp" or "utility creature"?
Creates ongoing "where does this go?" decisions for both humans and agents.
Rejected in favor of one-file-per-card which has zero ambiguity.

### Split by card type (artifacts.rs, creatures.rs, etc.)

Unambiguous categories but uneven distribution — creatures would be ~40% of all
cards. Doesn't solve the "one huge file" problem, just shifts it. Good as a
subdirectory structure at 2,000+ cards (under one-file-per-card), not good as a
file-level split.

### Split alphabetically

Zero ambiguity, even distribution, but no semantic grouping. A developer can't
open a file and see "all the equipment." Worse developer experience than
one-file-per-card where at least each file is self-documenting.

### One file per deck

Tied to specific test decks. Cards appearing in multiple decks have ambiguous
ownership. Doesn't generalize. Rejected.

### Data-driven (JSON/RON → codegen or runtime loading)

Loses compile-time type checking, IDE support, and refactoring safety. These are
most valuable during active engine development when the Effect DSL changes
frequently. Viable as a long-term option when the DSL stabilizes, but premature
now. See "Long-Term Option" section above.

## Implementation Plan

See `test-data/test-decks/PLAN.md` Phase 6 for the step-by-step refactoring plan.

### Execution order

1. **Worklist** — Build ability-gated classification script (no prereqs)
2. **Refactor** — Create `card-defs` crate, migrate 112 existing definitions
3. **Agent fix** — Update `card-definition-author` for new file structure
4. **Skeleton generator** — Scryfall → `.rs` scaffold script
5. **Batch authoring** — Start with Tier 2 ready cards, track performance

The worklist can start immediately. The refactor is the critical path for
steps 3-5.
