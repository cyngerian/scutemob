# MTG Game Script Generation & Validation Strategy

## Purpose

This document defines the strategy for generating engine-independent game scripts that serve
as a correctness oracle for the rules engine. These scripts are **not generated by the engine** —
they are generated by Claude Code reasoning directly from the Comprehensive Rules via MCP tools,
then validated by human review. Once validated, they become test fixtures that the engine must
match exactly.

This solves a fundamental testing problem: if the engine generates both the behavior and the
tests, bugs can hide. Game scripts are an independent specification of "what should happen,"
derived from the rules themselves rather than from engine code.

**This document is a reference for Claude Code during script generation and replay harness
implementation.**

---

## Core Workflow

```
1. SEED        Human or Claude Code defines a scenario (board state + intended actions)
                  │
2. GENERATE    Claude Code traces the scenario step-by-step using MCP tools
               to look up CR sections, card oracle text, and rulings.
               Outputs a JSON game script with CR citations at every step.
                  │
3. VALIDATE    Claude Code re-reads the script in a separate pass, independently
               checking each step's CR citations against the rules. Flags disputes.
                  │
4. REVIEW      Human reviews the script (especially disputed or complex steps).
               Marks confidence level. Approves or corrects.
                  │
5. REPLAY      Engine replay harness loads the script, feeds commands into
               the engine, and asserts state matches at every checkpoint.
                  │
6. VERDICT     Match = pass. Mismatch = investigate (engine bug or script bug).
```

**Key property**: Steps 1-4 never touch the engine. The script is correct because the rules
say so, not because the engine produces that result.

## Script JSON Schema

### Top-Level Structure

```json
{
  "schema_version": "1.0.0",
  "metadata": {
    "id": "script_deathtouch_trample_001",
    "name": "Deathtouch + Trample damage assignment in combat",
    "description": "A 5/5 with deathtouch and trample attacks and is blocked by a 2/2. Tests that only 1 damage is assigned to the blocker (deathtouch makes it lethal) and 3 tramples through.",
    "cr_sections_tested": ["702.2", "702.19c", "704.5h", "510.1"],
    "corner_case_ref": 8,
    "tags": ["combat", "deathtouch", "trample", "sba", "damage_assignment"],
    "confidence": "high",
    "review_status": "approved",
    "reviewed_by": "ian",
    "review_date": "2026-02-21",
    "generation_notes": "Generated by Claude Code using CR MCP tools. Cross-validated in separate pass.",
    "disputes": []
  },
  "initial_state": { },
  "script": [ ]
}
```

#### Metadata Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Unique identifier. Format: `script_<topic>_<nnn>` |
| `name` | string | yes | Human-readable name |
| `description` | string | yes | What this script tests and why it matters |
| `cr_sections_tested` | string[] | yes | All CR sections exercised by this script |
| `corner_case_ref` | number \| null | no | Reference to corner case number in `mtg-engine-corner-cases.md` |
| `tags` | string[] | yes | Searchable tags for filtering (subsystem, keywords, mechanics) |
| `confidence` | enum | yes | `"high"`, `"medium"`, `"low"` — see Confidence Levels below |
| `review_status` | enum | yes | `"pending_review"`, `"approved"`, `"disputed"`, `"corrected"` |
| `reviewed_by` | string \| null | no | Reviewer identifier |
| `review_date` | string \| null | no | ISO date of review |
| `generation_notes` | string | no | How the script was generated, any caveats |
| `disputes` | Dispute[] | no | Array of disputed steps with explanations (see below) |

#### Confidence Levels

- **high**: Simple interaction, well-documented in CR, no ambiguity. Auto-approvable after cross-validation pass.
- **medium**: Multiple subsystems interact, or rules text is ambiguous. Worth human review.
- **low**: Complex replacement effect chains, dependency resolution, or interactions not explicitly covered by CR examples. Requires human review before use.

#### Dispute Format

```json
{
  "step_index": 5,
  "action_index": 2,
  "raised_by": "cross_validation",
  "description": "Validator disagrees with damage assignment order. Generator assigned 1 to blocker citing CR 702.2, validator believes 2 is required citing CR 510.1c.",
  "resolution": "Generator correct — CR 702.2 explicitly states any amount is lethal with deathtouch.",
  "resolved_by": "ian",
  "resolved_date": "2026-02-21"
}
```

#### Initial State Guidelines

- **Include only what's relevant.** A script testing combat doesn't need to list every card
  in every player's library. Omitted zones default to empty. Omitted fields on permanents
  default to their natural state (untapped, no counters, no damage, not summoning sick).
- **Use card names, not IDs.** The replay harness resolves card names against the SQLite DB
  to get full oracle text and characteristics. Names must match Scryfall oracle card names exactly.
- **Specify all four players for Commander scripts.** Even if only two are involved in the
  interaction, the other two players exist and participate in priority passing.
- **Continuous effects must be listed explicitly** if the scenario depends on them. Don't
  assume the replay harness will know that a static ability is producing an effect — list it.
- **Libraries are unspecified by default.** If the script involves drawing or searching, provide
  the relevant cards at the top of the library in order.

### Script Actions

The `script` array is an ordered list of **steps**. Each step corresponds to a game step or
phase, and contains an ordered list of **actions** within that step.

```json
{
  "script": [
    {
      "step": "precombat_main",
      "step_note": "Alice's precombat main phase. She intends to cast Swords to Plowshares targeting Bob's Grizzly Bears.",
      "actions": [ ]
    },
    {
      "step": "beginning_of_combat",
      "actions": [ ]
    }
  ]
}
```

#### Step Identifiers

Valid `step` values correspond to the turn structure FSM in the engine:

```
"untap"
"upkeep"
"draw"
"precombat_main"
"beginning_of_combat"
"declare_attackers"
"declare_blockers"
"first_strike_damage"
"combat_damage"
"end_of_combat"
"postcombat_main"
"end_step"
"cleanup"
```

#### Action Types

Each action in a step is one of the following types:

##### `player_action` — A player takes a game action

```json
{
  "type": "player_action",
  "player": "alice",
  "action": "cast_spell",
  "card": "Swords to Plowshares",
  "targets": [
    {
      "type": "permanent",
      "card": "Grizzly Bears",
      "controller": "bob"
    }
  ],
  "mana_paid": { "W": 1 },
  "cr_ref": "601.2",
  "note": "Alice casts Swords to Plowshares targeting Bob's Grizzly Bears. Pays {W} from untapped Plains."
}
```

Other `action` values:
- `"cast_spell"` — cast from hand, command zone, or other zone
- `"activate_ability"` — activate an activated ability
- `"play_land"` — play a land for turn
- `"declare_attackers"` — declare attackers (see combat actions below)
- `"declare_blockers"` — declare blockers
- `"assign_damage"` — assign combat damage
- `"choose_option"` — choose a mode, target for a trigger, replacement effect ordering, etc.
- `"order_triggers"` — order simultaneously triggered abilities
- `"special_action"` — morph, unmorph, companion, etc.
- `"concede"` — player concedes
- `"mulligan_decision"` — keep or mulligan

##### `priority_pass` — A player passes priority

```json
{
  "type": "priority_pass",
  "player": "alice",
  "note": "Alice passes priority. Bob receives priority (next in turn order)."
}
```

##### `priority_round` — Shorthand for all players passing priority

When all players pass in succession with nothing happening, use this shorthand instead of
listing every individual pass. Only valid when the stack is empty or about to resolve.

```json
{
  "type": "priority_round",
  "players": ["alice", "bob", "carol", "dave"],
  "result": "all_pass",
  "note": "All players pass priority in APNAP order. Stack top resolves (or step advances if stack empty)."
}
```

##### `stack_resolve` — Top of stack resolves

```json
{
  "type": "stack_resolve",
  "object": "Swords to Plowshares",
  "resolution": [
    {
      "effect": "exile_permanent",
      "target": { "card": "Grizzly Bears", "controller": "bob" },
      "cr_ref": "608.2",
      "note": "Grizzly Bears is exiled."
    },
    {
      "effect": "gain_life",
      "player": "bob",
      "amount": 2,
      "cr_ref": "608.2",
      "note": "Bob gains life equal to Grizzly Bears' power (2)."
    }
  ],
  "note": "Swords to Plowshares resolves. Exile target creature, its controller gains life equal to its power."
}
```

##### `sba_check` — State-based actions are checked

```json
{
  "type": "sba_check",
  "results": [
    {
      "sba": "704.5f",
      "description": "Creature with 0 or less toughness is put into graveyard",
      "object": "Servo Token",
      "controller": "alice",
      "result": "moved_to_graveyard",
      "cr_ref": "704.5f"
    }
  ],
  "triggered_abilities": [
    {
      "trigger": "whenever a creature dies",
      "source": "Blood Artist",
      "controller": "bob",
      "cr_ref": "603.2"
    }
  ],
  "note": "SBAs checked. Servo Token has 0 toughness after -1/-1 effect, goes to graveyard. Blood Artist triggers."
}
```

When no SBAs apply:

```json
{
  "type": "sba_check",
  "results": [],
  "triggered_abilities": [],
  "note": "SBAs checked. No actions taken."
}
```

##### `trigger_placed` — Triggered ability goes on the stack

```json
{
  "type": "trigger_placed",
  "source": "Blood Artist",
  "controller": "bob",
  "trigger_condition": "whenever a creature dies",
  "targets": [],
  "ordering_context": "apnap",
  "apnap_position": 2,
  "cr_ref": "603.3b",
  "note": "Blood Artist's triggered ability is placed on the stack. Bob is second in APNAP order."
}
```

##### `assert_state` — Assert the current game state (checkpoint)

These are the verification points the replay harness checks against the engine.

```json
{
  "type": "assert_state",
  "description": "After Swords to Plowshares resolves",
  "assertions": {
    "players.bob.life": 40,
    "zones.battlefield.bob": {
      "excludes": ["Grizzly Bears"]
    },
    "zones.exile": {
      "includes": [{ "card": "Grizzly Bears", "owner": "bob" }]
    },
    "zones.graveyard.alice": {
      "includes": [{ "card": "Swords to Plowshares" }]
    },
    "zones.stack": {
      "is_empty": true
    }
  },
  "note": "Bob is at 40 life (was 38, gained 2). Grizzly Bears in exile. Swords in Alice's graveyard."
}
```

#### Assertion Path Syntax

Assertions use dot-notation paths into the game state:

| Path | What It Checks |
|------|---------------|
| `players.<name>.life` | Life total |
| `players.<name>.poison_counters` | Poison counter count |
| `players.<name>.commander_damage_received.<from>` | Commander damage from a specific player |
| `zones.battlefield.<player>` | Permanents a player controls |
| `zones.hand.<player>.count` | Number of cards in hand |
| `zones.graveyard.<player>` | Cards in graveyard |
| `zones.stack` | Current stack contents |
| `zones.stack.count` | Number of items on stack |
| `zones.exile` | All exiled cards |
| `permanent.<card_name>.tapped` | Whether a specific permanent is tapped |
| `permanent.<card_name>.counters.<type>` | Counter count on a permanent |
| `permanent.<card_name>.power` | Current power (after continuous effects) |
| `permanent.<card_name>.toughness` | Current toughness (after continuous effects) |
| `permanent.<card_name>.abilities` | List of abilities (after continuous effects) |
| `permanent.<card_name>.types` | Current types (after continuous effects) |
| `permanent.<card_name>.controller` | Current controller |

Assertion value types:
- **Exact match**: `"players.bob.life": 40`
- **Includes**: `"zones.graveyard.alice": { "includes": [{"card": "Lightning Bolt"}] }`
- **Excludes**: `"zones.battlefield.bob": { "excludes": ["Grizzly Bears"] }`
- **Is empty**: `"zones.stack": { "is_empty": true }`
- **Count**: `"zones.hand.alice.count": 3`
- **Comparison**: `"players.bob.life": { "less_than": 40 }`

##### `phase_transition` — The game advances to a new phase or step

```json
{
  "type": "phase_transition",
  "from_step": "precombat_main",
  "to_step": "beginning_of_combat",
  "cr_ref": "500.1",
  "note": "Main phase ends, combat phase begins."
}
```

##### `turn_based_action` — Automatic game action at the start of a step

```json
{
  "type": "turn_based_action",
  "action": "draw_card",
  "player": "alice",
  "cr_ref": "504.1",
  "note": "Alice draws a card for turn (draw step turn-based action)."
}
```

Other turn-based actions: `"untap_all"`, `"draw_card"`, `"empty_mana_pool"`, `"remove_until_eot"`, `"discard_to_hand_size"`

## Generating Scripts with Claude Code

### Generation Prompt Pattern

When generating a script, Claude Code should follow this process:

1. **Understand the scenario**: What board state, what actions, what interaction is being tested?

2. **Look up all relevant cards**: Use the card lookup MCP tool to get exact oracle text
   for every card in the scenario. Do not rely on memory — oracle text has been errataed
   many times.

3. **Look up all relevant rules**: Use the CR search MCP tool to find the exact rule text
   for every mechanic in the scenario. Trace the interaction step by step.

4. **Trace every game action in order**:
   - Who has priority?
   - What action do they take?
   - What goes on the stack?
   - Who gets priority next?
   - When do SBAs get checked?
   - What triggers fire?
   - In what order do triggers go on the stack (APNAP)?
   - When does each stack object resolve?
   - What is the state after resolution?

5. **Cite every step**: Every action in the script must reference the CR section that
   authorizes it. If you can't find a CR section, the step may be wrong.

6. **Insert assert_state checkpoints**: At minimum, assert state after every stack resolution
   and after every SBA check that produces results. More checkpoints are better — they
   narrow down where a mismatch occurs during replay.

7. **Output the JSON**: Follow the schema exactly. Validate field names and types.

### Cross-Validation Prompt Pattern

After generating a script, Claude Code should validate it in a separate reasoning pass:

```
Review the following game script. For each action:
1. Look up the cited CR section using the CR search tool
2. Verify the cited section supports the action taken
3. Verify the state assertions are consistent with the actions
4. Flag any step where the CR citation is wrong, missing, or doesn't support the asserted outcome

Focus especially on:
- Priority passing order (who gets priority after each action?)
- SBA check timing (are SBAs checked every time a player would receive priority?)
- Trigger ordering (APNAP in multiplayer)
- Zone change object identity (does the script correctly track new objects?)
```

## Implementation Hooks

These are the integration points where game scripts connect to the engine codebase.

### Hook 1: Script Schema as a Rust Type (M0)

Define the script schema as Rust types in a shared location so both the generator tool
and the replay harness use the same format.

**Location**: `crates/engine/src/testing/script_schema.rs` (or a shared crate if preferred)

```rust
use serde::{Deserialize, Serialize};
use std::collections::HashMap;

#[derive(Debug, Serialize, Deserialize)]
pub struct GameScript {
    pub schema_version: String,
    pub metadata: ScriptMetadata,
    pub initial_state: InitialState,
    pub script: Vec<ScriptStep>,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct ScriptMetadata {
    pub id: String,
    pub name: String,
    pub description: String,
    pub cr_sections_tested: Vec<String>,
    pub corner_case_ref: Option<u32>,
    pub tags: Vec<String>,
    pub confidence: Confidence,
```

### Hook 2: Replay Harness (M7)

The replay harness reads a `GameScript`, constructs the initial state using `GameStateBuilder`,
and then feeds each action as a `Command` to the engine. After each `assert_state` action,
it compares the engine's actual state to the expected assertions.

**Location**: `crates/engine/tests/script_replay.rs`

```rust
/// Pseudocode — actual implementation depends on engine API shape at M7
fn replay_script(script: &GameScript) -> Vec<ReplayResult> {
    let mut state = build_initial_state(&script.initial_state);
    let mut results = vec![];

    for (step_idx, step) in script.script.iter().enumerate() {
        for (action_idx, action) in step.actions.iter().enumerate() {
            match action {
                ScriptAction::PlayerAction { player, action, .. } => {
                    let command = translate_to_command(action, &state);
                    state = engine.process(state, command)
                        .expect("Engine rejected a command from a validated script");
                }
                ScriptAction::PriorityPass { player, .. } => {
                    let command = Command::PassPriority { player: resolve_player(player) };
                    state = engine.process(state, command).unwrap();
                }
                ScriptAction::AssertState { assertions, description, .. } => {
                    let mismatches = check_assertions(&state, assertions);
                    if !mismatches.is_empty() {
                        results.push(ReplayResult::Mismatch {
                            step_idx,
                            action_idx,
                            description: description.clone(),
                            mismatches,
                        });
                    } else {
                        results.push(ReplayResult::Match {
                            step_idx,
                            action_idx,
                        });
                    }
                }
                ScriptAction::SbaCheck { results: expected_results, .. } => {
                    // Engine should have already run SBAs as part of the priority
                    // system. Verify that the SBA results match expectations.
                    // This may require the engine to emit SBA events that we can inspect.
                }
                // ... handle other action types
                _ => {}
            }
        }
    }
    results
}
```

**Critical design note**: The replay harness must be able to compare the engine's SBA results
against the script's expected SBA results. This means the engine must emit structured events
for SBAs, not just silently modify state. This aligns with the command/event architecture
in `mtg-engine-architecture.md` Section 4.1 — SBAs produce `GameEvent` values that the
replay harness can inspect.

### Hook 3: Test Discovery (M7+)

Automatically discover and run all approved scripts:

**Location**: `crates/engine/tests/run_all_scripts.rs`

```rust
use std::fs;
use std::path::Path;

fn discover_scripts(dir: &Path) -> Vec<GameScript> {
    fs::read_dir(dir)
        .unwrap()
        .filter_map(|entry| {
            let path = entry.ok()?.path();
            if path.extension()? == "json" {
                let content = fs::read_to_string(&path).ok()?;
                let script: GameScript = serde_json::from_str(&content).ok()?;
                // Only run approved scripts
                if script.metadata.review_status == ReviewStatus::Approved {
                    Some(script)
                } else {
                    None
                }
            } else {
                None
            }
        })
        .collect()
}

#[test]
fn run_all_approved_scripts() {
    let scripts = discover_scripts(Path::new("../../test-data/generated-scripts"));
    assert!(!scripts.is_empty(), "No approved scripts found");

    let mut failures = vec![];
    for script in &scripts {
        let results = replay_script(script);
        let mismatches: Vec<_> = results.iter()
            .filter(|r| matches!(r, ReplayResult::Mismatch { .. }))
            .collect();
        if !mismatches.is_empty() {
            failures.push((script.metadata.id.clone(), mismatches));
        }
    }

    if !failures.is_empty() {
        for (id, mismatches) in &failures {
            eprintln!("SCRIPT FAILED: {}", id);
            for m in mismatches {
                eprintln!("  {:?}", m);
            }
        }
        panic!("{} of {} scripts failed", failures.len(), scripts.len());
    }
}
```

## Versioning and Schema Evolution

The `schema_version` field in every script tracks compatibility. When the schema changes:

1. Bump `schema_version` (semver)
2. Update the Rust types in `script_schema.rs`
3. Write a migration script if existing scripts need updating (or re-generate them)
4. The replay harness should reject scripts with unsupported schema versions with a clear error

**Breaking changes** (major version bump): Adding required fields, changing action type names,
restructuring the initial state format.

**Non-breaking changes** (minor version bump): Adding optional fields, adding new action types,
adding new assertion path types.
