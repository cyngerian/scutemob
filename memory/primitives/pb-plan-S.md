# Primitive Batch Plan: PB-S — GrantActivatedAbility / GrantManaAbility

**Generated**: 2026-04-11
**Primitive**: Static continuous effect that grants a specific activated (or mana) ability to a filtered set of permanents via Layer 6 (CR 613.1f).
**CR Rules**: 112 (abilities), 602 (activated abilities), 605.1a (mana abilities), 611 (continuous effects), 613.1f (Layer 6), 613.5 (stacking), 706 (copies), 302.1 (summoning sickness)
**Cards affected**: 6 existing TODOs fixed + 0 new defs. Song of Freyalise and Umbral Mantle partially unblocked (Saga/{Q} still gate them).
**Dependencies**: None new. Builds on existing `AbilityDefinition::Static { continuous_effect }` + `LayerModification` + `calculate_characteristics`.
**Deferred from prior PBs**: W3-LC Medium bugs "abilities.rs:222,246 — activated ability access bypasses Humility" — already FIXED in `handle_activate_ability` (uses `calculate_characteristics`). `handle_tap_for_mana` and `legal_actions` still read base — PB-S must fix them as part of the layer-wiring.

---

## Summary

Add two new `LayerModification` variants — `AddActivatedAbility(ActivatedAbility)` and
`AddManaAbility(ManaAbility)` — that let a static ability grant a single specific
ability to every permanent matching an `EffectFilter`. Reuses the existing
`AbilityDefinition::Static { continuous_effect: ContinuousEffectDef { layer: Ability, ... } }`
registration path (no new ability variant). Fix the residual base-reads of
`mana_abilities`/`activated_abilities` in `rules/mana.rs` and `simulator/legal_actions.rs`
so granted abilities flow through discovery and dispatch.

This is **narrow**: one specific ability per grant. Does NOT handle reflection
("gains all activated abilities of another permanent" — Marvin), ability REMOVAL,
nor text-changing effects. Scope boundary enforced by naming the variants after
what they do ("Add one ability"), not "Grant arbitrary abilities".

## CR Citations

- **CR 112.3**: Characteristics modified by continuous effects.
- **CR 602**: Activated abilities, cost + effect, paid/resolved by the permanent itself.
- **CR 605.1a**: A granted mana-adding tap ability that requires no target is a **mana ability**
  regardless of source of the grant. Cryptolith Rite / Chromatic Lantern / Citanul Hierophants /
  Paradise Mantle / Enduring Vitality / Song of Freyalise all grant mana abilities per this rule —
  therefore they do NOT use the stack, resolve immediately via `handle_tap_for_mana`.
- **CR 611.3a**: A continuous effect generated by a static ability's resolution is applied
  whenever the ability is in effect (i.e., its source is on the battlefield).
- **CR 613.1f**: **Layer 6** — ability-adding, ability-removing, and "can't have" effects.
  All PB-S effects apply in `EffectLayer::Ability`.
- **CR 613.5**: Two separate grant sources produce two separate ability instances on the
  target (they don't "collapse"). With the Cryptolith Rite ruling (2016-04-08), tapping
  a creature with two grants only satisfies ONE `{T}` cost, so only one grant activates per tap.
- **CR 706**: Copiable values include printed abilities and abilities granted by the card's own
  resolution effects, but **not** abilities granted to it by other permanents' static abilities.
  Relevant for: a creature under Cryptolith Rite that is copied does NOT keep the grant unless
  the copy is also on the battlefield with Cryptolith Rite (the grant is re-applied from layers).
- **CR 302.1 / 302.6**: A creature granted `{T}: Add mana` still suffers summoning sickness
  unless it has haste — already enforced in `handle_tap_for_mana` (lines 138-159) via
  `calculate_characteristics`.
- **CR 400.7**: When a permanent leaves the battlefield, the grant is gone (layer effect ends
  when source leaves via `EffectDuration::WhileSourceOnBattlefield`). Standard for static abilities.

**Chromatic Lantern ruling (2018-10-05)**: "Lands you control won't lose any other abilities
they had. They also won't gain or lose any land types." → grant is **additive**; the layer
modification must APPEND to `chars.mana_abilities`, not replace.

**Cryptolith Rite ruling (2016-04-08)**: Two Rites give a creature two identical mana
abilities, but the single `{T}` cost can only satisfy one activation. Implementation falls
out naturally: each grant source pushes one ability entry; the tap cost prevents both from
firing on one tap.

## Design Decision: Layer 6 injection vs parallel registry

**Chosen: Layer 6 injection.** `calculate_characteristics` already populates the Ability
layer; two new `LayerModification` variants push directly onto
`chars.activated_abilities` / `chars.mana_abilities`. The alternative (parallel registry
à la `flash_grants`, `etb_suppressors`) would require every ability dispatch site to
walk two lists and concatenate virtual indices — error-prone.

This choice forces us to fix two stale base-reads that the W3-LC audit already flagged,
but those are shallow: one-line `obj.characteristics.X` → `calculate_characteristics(state, id).unwrap_or(obj.characteristics.clone()).X`.

`handle_activate_ability` (abilities.rs:220, 295) **already** resolves through calc chars, so
non-mana granted activated abilities work out of the box once the LayerModification is
implemented.

///
/// The ability is appended to `chars.activated_abilities`. Multiple grant sources
/// produce multiple ability entries (CR 613.5). The ability's source is always the

/// granted. It resolves immediately without using the stack.

/// Enduring Vitality, Song of Freyalise (chapters I/II).

## Edge Cases & Risks

1. **Grant stacking (two Cryptolith Rites)**: Each static ability pushes one entry onto `chars.activated_abilities`/`mana_abilities`. The vec holds two identical entries. `handle_tap_for_mana` taps the creature at activation of index 0 → now tapped → activation of index 1 fails on `already_tapped`. Works by construction. **Test**: `test_two_cryptolith_rites_grant_two_abilities_but_one_tap`.

2. **Grant removal mid-turn**: Standard layer system — when Rite leaves, `WhileSourceOnBattlefield` stops the effect, next `calculate_characteristics` call omits the grant. Any already-tapped creature stays tapped. `abilities_activated_this_turn` counter stays. **Test**: `test_cryptolith_rite_grant_ends_when_source_leaves`.

3. **Attached vs global**: Paradise Mantle uses `EffectFilter::AttachedCreature` (already resolves via `obj.attached_to` at characteristic time, layers.rs:594). Cryptolith Rite uses `CreaturesYouControl`. Both filters are pre-existing and correct.

4. **Copy effects (CR 706)**: A creature granted `{T}: Add mana` by Cryptolith Rite has the ability **as a continuous effect**, not as a copiable value. Copying the creature does NOT copy the grant; however, if the copy is on the battlefield, Cryptolith Rite's layer effect applies to it equally. Result: same outcome. No test needed beyond documenting in plan.

5. **Haste / summoning sickness (CR 302.1)**: Enforced in `handle_tap_for_mana` lines 138-159 via `calculate_characteristics` (already layer-correct). **Test**: `test_granted_mana_ability_respects_summoning_sickness`.

6. **Humility interaction (CR 613.1f)**: Humility = `LayerModification::RemoveAllAbilities`. If Humility's timestamp is later than Cryptolith Rite's, it wipes the grant (it reads `chars.mana_abilities = Vector::new()`). If earlier, Rite re-adds the grant on top. Timestamp determines order — test with later Humility to confirm wipe. Note: `RemoveAllAbilities` also wipes keywords. Current layers.rs code clears `chars.mana_abilities` in `RemoveAllAbilities` (verified L1061), so Humility after Rite correctly strips the grant. **Test**: `test_humility_removes_granted_mana_ability`.

7. **LegalActionProvider mana_solver path**: The simulator bot's mana-solver (`crates/simulator/src/mana_solver.rs`) enumerates mana sources to find how to afford a cost. If it reads base `mana_abilities`, it will miss granted sources → bots play suboptimally but not incorrectly. **Not blocking PB-S**; document as LOW follow-up.

8. **Replay determinism (hash)**: Adding `ActivatedAbility` / `ManaAbility` to layer effects means they participate in `HashInto` for `LayerModification`. Confirm these types hash deterministically (OrdMap/OrdSet/Vec in canonical order) — should already, since `GameObject` is hashable.

9. **Ability index stability**: `handle_activate_ability` takes an integer `ability_index` into the calculated `activated_abilities` vec. If a grant appears/disappears between the player seeing legal actions and submitting the command, the index may shift. **Risk is pre-existing** (layer effects can already change between observation and command); PB-S doesn't worsen it. Document only.

10. **Granted activation sacrifice/sacrifice_self**: None of the target cards use this. If a future grant included sacrifice-self (e.g. "equipped creature has '{T}, sac this: ...'"), the sac would apply to the host, not the grantor — correct semantically but worth noting. Not tested in PB-S.

## Open Questions for Oversight

1. **Should grant LayerModification also wipe `chars.abilities` (AbilityInstance)?** The engine
   has a parallel `abilities: Vector<AbilityInstance>` field alongside the specialized ones.
   Unclear if this is used for dispatch. Recommend: **no** — keep PB-S to the two specialized
   vecs and leave AbilityInstance to whoever owns it. Flag if implementation finds a contradiction.

2. **Do we need a test for "grant interaction with Cloak/face-down creatures"?** Face-down
   creatures lose their printed abilities but remain subject to static effects. A face-down
   creature under Cryptolith Rite should gain the mana ability. Low priority; add if trivial.

3. **Hash version bump?** Adding new LayerModification discriminants changes the hash space.
   If hash versioning is tracked for save-file compatibility, bump. If not, no-op. Implementation
   runner should check `memory/gotchas-infra.md` for hash-version policy.

4. **mana_solver.rs fix scope**: include in PB-S or defer? Recommend defer — PB-S is already
   wide enough, and bot behavior is non-correctness.
