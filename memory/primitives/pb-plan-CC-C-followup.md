## Step 1 — CR research (verbatim from MCP rules server)

### CR 611.3 — continuous effects from static abilities

> 611.3. A continuous effect may be generated by the static ability of an object.
>
> [611.3a] A continuous effect generated by a static ability isn't "locked in"; it applies at any given moment to whatever its text indicates.
>
> [611.3b] The effect applies at all times that the permanent generating it is on the battlefield or the object generating it is in the appropriate zone.
>
> [611.3c] Continuous effects that modify characteristics of permanents do so simultaneously with the permanent entering the battlefield. They don't wait until the permanent is on the battlefield and then change it. Because such effects apply as the permanent enters the battlefield, they are applied before determining whether the permanent will cause an ability to trigger when it enters the battlefield.
>
> [611.3d] Continuous effects from static abilities may allow a player to play a land or cast a permanent spell, or may grant an ability to a permanent spell or card that allows it to be cast. […]

### CR 613.4 — Layer 7 sublayers

> 613.4. Within layer 7, apply effects in a series of sublayers in the order described below. Within each sublayer, apply effects in timestamp order. (See rule 613.7.) Note that dependency may alter the order in which effects are applied within a sublayer. (See rule 613.8.)
>
> [613.4a] Layer 7a: Effects from characteristic-defining abilities that define power and/or toughness are applied. See rule 604.3.
>
> [613.4b] Layer 7b: Effects that set power and/or toughness to a specific number or value are applied. Effects that refer to the base power and/or toughness of a creature apply in this layer.
>
> [613.4c] Layer 7c: Effects and counters that modify power and/or toughness (but don't set power and/or toughness to a specific number or value) are applied.
>
> [613.4d] Layer 7d: Effects that switch a creature's power and toughness are applied. […]

### CR 604.2 / 604.3 / 604.3a — static abilities create continuous effects; CDA criteria

> 604.2. Static abilities create continuous effects, some of which are prevention effects or replacement effects. These effects are active as long as the permanent with the ability remains on the battlefield and has the ability, or as long as the object with the ability remains in the appropriate zone, as described in rule 113.6.
>
> 604.3. Some static abilities are characteristic-defining abilities. […]
>
> [604.3a] A static ability is a characteristic-defining ability if it meets the following criteria: (1) It defines an object's colors, subtypes, power, or toughness; (2) it is printed on the card it affects, it was granted to the token it affects by the effect that created the token, or it was acquired by the object it affects as the result of a copy effect or text-changing effect; (3) it does not directly affect the characteristics of any other objects; (4) it is not an ability that an object grants to itself; and (5) it does not set the values of such characteristics only if certain conditions are met.

### CR 608.2h — spell-effect X locked at resolution (the contrast rule)

> 608.2h. If an effect requires information from the game (such as the number of creatures on the battlefield), the answer is determined only once, when the effect is applied. If the effect requires information from a specific object, including the source of the ability itself, the effect uses the current information of that object if it's in the public zone it was expected to be in […]

### CR 122.1 + Vishgraz 2023-02-04 ruling — sum semantic

> 122.1. A counter is a marker placed on an object or player that modifies its characteristics or interacts with an ability. […]

Vishgraz 2023-02-04 ruling (per PB-CC-A pb-review-CC-A.md and `crates/engine/src/cards/defs/vishgraz_the_doomhive.rs:53-55`): "Each opponent" sums poison counters across opponents (3 opponents with 1, 2, 5 = 8, NOT count-of-poisoned-opponents = 3 and NOT max = 5).

### Path-semantics contrast (load-bearing for this PB)

- **CR 608.2h**: spell-effect X is locked at the moment the spell resolves. Implemented today by `effects/mod.rs:2330-2347` (substitution arm). Vishgraz/Fuseling cannot use this path — their P/T must update as opponents acquire poison / counters move.
- **CR 611.3a**: static-ability continuous effect is *not* locked; it applies to whatever its text indicates "at any given moment." This is what Vishgraz and Fuseling need. Implemented today only at Layer 7a via `SetPtDynamic` + `resolve_cda_amount` (`layers.rs:1122-1132, 1429-1592`). Layer 7c has *no* live-eval path — that is the gap this PB fills.

### Site 2 — `register_static_continuous_effects` in `crates/engine/src/rules/replacement.rs:1753-1797`

Currently iterates `AbilityDefinition::Static { continuous_effect }` and pushes a `ContinuousEffect` with **`is_cda: false`** hardcoded at line 1795. The dynamic-static path needs `is_cda: true` for CR 604.3a-compliant registration AND to flow through the live-eval branch in `apply_modification` (Site 4).

**Required change**: introduce a *new* `AbilityDefinition` variant — `CdaModifyPowerToughness { power: Option<EffectAmount>, toughness: Option<EffectAmount> }` — registered analogously to the existing `CdaPowerToughness` arm at lines 1832-1857, but at Layer 7c (`EffectLayer::PtModify`) instead of Layer 7a (`EffectLayer::PtCda`). Each `Some(amount)` produces one Layer-7c continuous effect with the corresponding `ModifyPowerDynamic` / `ModifyToughnessDynamic` / `ModifyBothDynamic` modification, `is_cda: true`, `EffectFilter::SingleObject(new_id)`, `WhileSourceOnBattlefield` duration, `condition: None`. The match treats `Some(p)+Some(t)` with the same EffectAmount as `ModifyBothDynamic` for hash compactness, otherwise uses two separate single-axis effects.

**Why a new AbilityDefinition variant rather than overloading `Static`**: per `feedback_verify_full_chain.md`, type-system enforcement is preferred. `Static { continuous_effect }` allows arbitrary modifications and would need a runtime check to forbid the dynamic variants; the new `CdaModifyPowerToughness` variant constrains authors to amounts only and matches CR 604.3a(5)'s no-condition rule structurally. Mirrors `CdaPowerToughness` (Layer 7a) precedent exactly.

## Step 3 — Shape decision

**Chosen: Shape A + Shape D hybrid** — a new `AbilityDefinition` variant + re-resolve at `apply_modification` time using the existing `resolve_cda_amount` helper. This is exactly the Layer-7a precedent transposed to Layer 7c.

### Why not Shape B (`is_static: bool` flag on existing variants)?

Adding a flag means *every* call site that constructs `ModifyPowerDynamic` must pick the right flag, and the substitution arm must check the flag before substituting. This invites authoring bugs ("I forgot the flag"), exactly the kind of footgun PB-CC-C explicitly added doc-comments to prevent. Type-system enforcement (per `feedback_verify_full_chain.md`) is preferred.

### Why not pure Shape C (route via apply_modification with no register-side changes)?

This works mechanically but loses the AbilityDefinition-level type discipline. Authors would still have to write `AbilityDefinition::Static { continuous_effect: ContinuousEffectDef { modification: ModifyPowerDynamic, ... } }` — exactly the path the PB-CC-C doc-comments warn against. The CR-604.3a(5)-no-condition invariant is also harder to enforce structurally without a dedicated variant.

### Why not pure Shape A (new LayerModification variants like `ModifyBothDynamicStatic`)?

Variant proliferation. We already have three dynamic variants (Power/Toughness/Both); doubling to six (with `…Static` suffix) means six hash arms, six panic-guard arms in `apply_layer_modification`, six substitution arms, etc. The `is_cda: true` flag on the existing `ContinuousEffect` already discriminates the path (substituted vs. live-eval). Reusing it is parsimonious.

### Chosen shape — what each dispatch site sees

| Site | Spell-effect path (CR 608.2h) | Static-ability path (CR 611.3a) |
|---|---|---|
| **Author DSL** | `Effect::ApplyContinuousEffect { effect_def: { modification: ModifyPowerDynamic { amount, negate: false }, … } }` (resolves via spell, locks at resolution) | `AbilityDefinition::CdaModifyPowerToughness { power: Some(amount), toughness: None }` (NEW VARIANT — register at ETB, evaluate at every layer pass) |
| **Storage** | `ContinuousEffect { modification: ModifyPower(N), is_cda: false, … }` | `ContinuousEffect { modification: ModifyPowerDynamic { amount, negate: false }, is_cda: true, … }` |
| **Layer apply** | Existing `ModifyPower(d)` arm at `layers.rs:1144-1148` | NEW: `ModifyPowerDynamic { amount, negate }` arm calls `resolve_cda_amount(state, amount, object_id, controller)`, applies negation, mutates `chars.power` |
| **Hash** | Existing arm 9/10/18 (concrete `ModifyPower(N)`) | Existing arm 26/27/25 (dynamic variant); new `AbilityDefinition::CdaModifyPowerToughness` arm |

### How does the substitution arm know NOT to substitute static-ability dynamic mods?

Because the substitution arm is only reached via `Effect::ApplyContinuousEffect` (executed during spell/ability resolution by `execute_effect`). Static abilities are registered by `register_static_continuous_effects` at ETB time, which never calls `execute_effect` — it directly pushes a `ContinuousEffect` into `state.continuous_effects`. The two paths are statically disjoint. PB-CC-C T5 (`test_modify_power_dynamic_x_locked_at_resolution`) tests the spell path with no static-ability registration; its assertion that "stored effect is `ModifyPower(N)` not `ModifyPowerDynamic`" remains true because the test calls `execute_effect` directly.

## Risks & edge cases

1. **Substitution arm pre-empts static path** — `Effect::ApplyContinuousEffect` is the spell-effect entrypoint. If a card def accidentally routes a static ability through it, the dynamic value gets locked at ETB time (CR 608.2h trap). The `AbilityDefinition::CdaModifyPowerToughness` variant is the structural prevention; no spell-effect path needs to know about it.
2. **`resolve_cda_amount` recursion** — Layer-7c CDA effects evaluate during `apply_layer_modification`, which runs inside `calculate_characteristics`. If `resolve_cda_amount` were to call `calculate_characteristics` (e.g. via `PowerOf` / `ToughnessOf`), infinite recursion. Mitigation: `resolve_cda_amount` already uses **base characteristics** (line 1453 comment) for filter checks, and the supported variants for the new branch (`Fixed`, `CounterCount`, `PermanentCount`, `CardCount`, `DevotionTo`, `PlayerCounterCount`, `Sum`, `DomainCount`) read base characteristics or non-layered fields. `XValue` / `LastEffectCount` / `LastDiceRoll` fall through to the `_ =>` panic at lines 1584-1591 — verify this behavior is preserved.
3. **CDA with Layer 4 type-changing dependency** — Vishgraz/Fuseling don't depend on type changes, so this is theoretical. But: if a future card uses `PermanentCount { filter: { types: Goblin } }` in a Layer-7c CDA and a Layer-4 effect adds the Goblin type, the layer order matters. Layer 4 < Layer 7c, so layer 4 has already been applied by the time we hit Layer 7c. But `resolve_cda_amount` reads base characteristics, not layer-resolved characteristics, so the dynamic count would NOT see the Layer-4 type addition. Document as a known limitation in the new variant's doc-comment (matches the existing `DomainCount` limitation comment at `layers.rs:1556-1558`).
4. **Negation on poison counters** — Vishgraz's "+1/+1 for each poison counter" never has `negate: true`, but the variant supports it. If a future card uses `negate: true` with `PlayerCounterCount`, the formula is `chars.power -= raw`. Test (b) only tests `negate: false`; consider an additional negation test if a future card needs it (out of scope for this PB).
5. **Sentinel test rename hygiene** — `pbt_up_to_n_targets.rs::test_pbt_hash_schema_version_is_12` was already stale per PB-CC-A T1 (LOW). Rename to `…_is_13` and update the docstring; closes the LOW concurrently with this PB.
6. **Hash arm choice for `Option<EffectAmount>`** — `Option<T>` has a canonical hash via `HashInto for Option<T>` (verify exists in `hash.rs`); if not, hash a 0u8 for `None`, 1u8 + inner for `Some`. Standard pattern.
7. **Replay-viewer / TUI exhaustive matches** — adding a new `AbilityDefinition` variant may break exhaustive matches in `tools/replay-viewer/src/view_model.rs` and `tools/tui/src/play/panels/stack_view.rs`. Per `MEMORY.md` ("Behavioral Gotchas — replay-viewer view_model.rs"), **runners miss this ~50% of the time**. Verify with `cargo build --workspace` (NOT just `cargo build`) after impl.
8. **Concurrent counter changes during layer pass** — `calculate_characteristics` takes `&GameState`; counters are not mutated mid-pass. Determinism preserved.
