## Primitive Specification

`EffectDuration` (`crates/card-types/src/state/continuous_effect.rs` L43-64) gains a sixth
variant:

```rust
/// Active for as long as the effect's creator controls the source permanent (CR 611.2b/c).
/// The `PlayerId` is "you" — captured at effect *creation* (the resolving spell/ability's
/// controller). Differs from `WhileSourceOnBattlefield` only under a control change of the
/// source: when the source's controller stops equalling this PlayerId, the effect ends
/// PERMANENTLY (CR 611.2c: the set of affected objects is fixed and the effect does not
/// resume if control later returns). Removal + control reversion is handled imperatively by
/// `expire_while_you_control_source_effects` (layers.rs); this arm of `is_effect_active`
/// therefore returns `true` (mirrors UntilYourNextTurn/UntilEndOfTurn, which are also
/// removed by an explicit pass rather than a live check).
///
/// Placeholder convention: card DSL literals write `WhileYouControlSource(PlayerId(0))`;
/// the `Effect::GainControl` / `Effect::ApplyContinuousEffect` handlers replace `PlayerId(0)`
/// with `ctx.controller` at execution time (identical to `UntilYourNextTurn`).
WhileYouControlSource(PlayerId),
```

**Why a new variant and not a reuse of `WhileSourceOnBattlefield` + a condition:** the two
durations are identical while the source stays under its original controller; they diverge on
exactly one event — control of the source changing away. A `Condition`-based approach would be
a **live re-evaluation**, which would incorrectly *revive* the effect if control returned
(violating CR 611.2c). The correctness requirement ("never resumes") forces a one-shot
imperative expiry, which is why this is a distinct duration with a distinct expiry pass.

## CR Rule Text (from MCP, authoritative)

**611.2b** — "Some continuous effects generated by the resolution of a spell or ability have
durations worded 'for as long as . . . .' If the 'for as long as' duration never starts, the
effect does nothing. Similarly, if that duration ends before the moment the effect would first
be applied and doesn't begin again during that spell or ability's resolution, the effect does
nothing. It doesn't start and immediately stop again, and it doesn't last forever."

**611.2c** — "If a continuous effect generated by the resolution of a spell or ability
modifies the characteristics or changes the controller of any objects, the set of objects it
affects is determined when that continuous effect begins. After that point, the set won't
change. (Note that this works differently than a continuous effect from a static ability.) …"

→ This is the "never resumes" property: because the affected set (and the "you" reference) is
**fixed at the beginning of the effect**, once the "for as long as you control [source]"
duration ends, the effect is over. It does not re-latch onto the object if the creator regains
control of the source. Combined with 611.2b ("it doesn't last forever"), the borrowed permanent
returns to its owner and stays there.

**613.1b** — control-changing effects are Layer 2. (`Effect::GainControl` already builds a
Layer-2 `SetController` continuous effect.)

**702.26e** — a phased-out permanent is treated as though it does not exist for most rules
purposes, **but its controller does not change** (it phases in under the same control). Load-
bearing here: a phased-out source must **not** end a `WhileYouControlSource` effect — you still
control a phased-out permanent. This is the one place where the new duration's activity test
must deliberately diverge from the `WhileSourceOnBattlefield` arm (layers.rs L503-509), which
treats a phased-out source as inactive.

## Engine finding established during research (drives the design)

**There is no control-reversion mechanism in the engine today.** Traced definitively:

- `Effect::GainControl` (effects/mod.rs L5356-5385) writes `obj.controller` **imperatively**
  (L5381) and pushes a Layer-2 `SetController` continuous effect.
- `calculate_characteristics` treats `LayerModification::SetController(_)` as a **no-op**
  (layers.rs L1092-1096, "controller lives on GameObject, not Characteristics") — there is no
  layer-driven control application.
- No reconcile loop consumes `SetController` effects. Grep of the whole engine for
  `SetController` / `EffectLayer::Control` reveals only: the GainControl + ExchangeControl
  writers (effects/mod.rs), the layer no-op (layers.rs L1094), and the hash arm (hash.rs L2020).
- `expire_end_of_turn_effects` (layers.rs L1576) and `expire_until_next_turn_effects`
  (layers.rs L1624) `retain` the effect **out** of `continuous_effects` but **never touch
  `obj.controller`**. The existing test `test_gain_control_until_eot_expires`
  (primitive_pb32.rs L333-374) only asserts the *effect* is removed, not that control reverts.
- `move_object_to_zone` (state/mod.rs L1097) resets `controller = owner` (L1150) **only for
  the object that changes zone** — not for a borrowed permanent whose *source* leaves.

**Consequence:** `WhileSourceOnBattlefield` gain-control (olivia, dragonlord_silumgar) does
**not** actually revert control when the source leaves — contrary to what
`dragonlord_silumgar.rs`'s note L54-57 claims ("control correctly reverts when Silumgar
leaves"). That note is **false**. Likewise `UntilEndOfTurn` gain-control (sarkhan_vol,
zealous_conscripts, karrthus_tyrant_of_jund) never reverts at cleanup — a latent legal-but-
wrong gap (see OOS-EF9-1 below).

So the instruction "revert control the same way the existing one does" cannot be followed
literally — the existing one does not revert at all. **PB-EF9 must build reversion**, and
because control is imperative, the reversion must be imperative too: recompute
`obj.controller` from owner + still-active `SetController` effects. The flip of olivia/silumgar
to the new duration is therefore a **strict correctness improvement on both axes** (source
leaves AND source stolen), since the new expiry handles the source-gone case via the same code.

- [x] **File**: `crates/card-types/src/state/continuous_effect.rs`, in the `EffectDuration`

### Change 2: `is_effect_active` arm

- [x] **File**: `crates/engine/src/rules/layers.rs`, `is_effect_active` (L501-561). Add:
  ```rust
  // CR 611.2b/c: activity/removal is handled imperatively by
  // expire_while_you_control_source_effects (one-shot, never resumes). Mirror
  // UntilYourNextTurn: always "active" here; the expiry pass owns termination.
  EffectDuration::WhileYouControlSource(_) => true,
  ```
  **Do NOT** put the control check here — a live check would let the effect resume, violating
  CR 611.2c. (`SetController` is a layer no-op anyway, so this arm never affects characteristics.)

### Change 3: placeholder → controller resolution at effect creation

- [x] **File**: `crates/engine/src/effects/mod.rs`, `Effect::GainControl` handler (L5356-5385).
  Currently `duration: *duration` (L5373). Replace with a resolved duration computed before the
  effect literal (mirror the `UntilYourNextTurn` pattern at L3147-3157):
  ```rust
  let resolved_duration = match *duration {
      EffectDuration::WhileYouControlSource(_) =>
          EffectDuration::WhileYouControlSource(controller), // controller == ctx.controller (L5358)
      other => other,
  };
  ```
  and use `duration: resolved_duration`. **CR 611.2c** — "you" is fixed to the creator at
  creation. This is the path olivia and dragonlord_silumgar use (both authored via `GainControl`).
- [x] **File**: `crates/engine/src/effects/mod.rs`, `Effect::ApplyContinuousEffect` handler,
  the `resolved_duration` match (L3150-3157). Add a `WhileYouControlSource(_) =>
  WhileYouControlSource(ctx.controller)` arm alongside the existing `UntilYourNextTurn` arm.
  (Covers any future card that authors the duration via `ApplyContinuousEffect`; roil_elemental
  will use `GainControl`, but keep the two creation paths symmetric.)
- [x] **File** (optional, for symmetry): `Effect::ExchangeControl` handler (effects/mod.rs
  L5387-5438) also uses `duration: *duration`. No current card needs `WhileYouControlSource`
  here; add the same resolution arm if trivial, otherwise leave a `// NOTE:` that
  ExchangeControl does not yet resolve the placeholder. Do not block the PB on this.
  **Deviation**: left as a `// NOTE:` comment (not a resolution arm) since no card uses this
  combination — matches the plan's "otherwise" branch.

### Change 4: the expiry + reversion pass (the core of the PB)

- [x] **File**: `crates/engine/src/rules/layers.rs`, new `pub fn`
  `expire_while_you_control_source_effects(state: &mut GameState)` next to the other `expire_*`
  functions (after L1671). Logic:
  1. Collect ended effects: for each `e` in `state.continuous_effects` where
     `matches!(e.duration, EffectDuration::WhileYouControlSource(_))`, compute `ended`:
     ```rust
     let EffectDuration::WhileYouControlSource(pid) = e.duration else { unreachable };
     let ended = match e.source {
         Some(src) => state.objects.get(&src)
             // CR 702.26e: phased-out source is STILL controlled — do NOT check is_phased_in().
             // Ended iff source is no longer a battlefield permanent OR its controller != pid.
             .map(|o| !(o.zone == ZoneId::Battlefield && o.controller == pid))
             .unwrap_or(true),           // source object gone (CR 400.7 new id) → ended
         None => false,
     };
     ```
     Record `(e.id, affected_object_ids)` where `affected_object_ids` comes from `e.filter`
     (for `GainControl` it is always `EffectFilter::SingleObject(id)`; handle `SingleObject`;
     ignore other filters — control effects from GainControl are always SingleObject).
  2. **Permanently** remove the ended effects: rebuild `state.continuous_effects` retaining
     those whose `id` is not in the ended set. This is the "never resumes" guarantee — nothing
     re-adds them (CR 611.2c).
  3. For each affected object id, call `recompute_object_controller(state, id)` (Change 5).
  4. Return `Vec<GameEvent>` if a control-change event exists (see Change 6); otherwise return
     nothing / `()`.

### Change 5: control-recompute helper (correct under stacked control)

- [x] **File**: `crates/engine/src/rules/layers.rs`, new
  `fn recompute_object_controller(state: &mut GameState, object_id: ObjectId)`:
  1. `let owner = state.objects.get(&object_id)?.owner;`
  2. Collect all **still-active** Layer-2 `SetController` effects whose filter is
     `SingleObject(object_id)`: `e.layer == EffectLayer::Control && is_effect_active(state, e)
     && e.filter == SingleObject(object_id) && matches!(e.modification, SetController(_))`.
     Sort by `e.timestamp` (CR 613.7).
  3. `let mut controller = owner; for e in sorted { if let SetController(p) = e.modification
     { controller = p; } }`
  4. `if let Some(o) = state.objects.get_mut(&object_id) { o.controller = controller; }`
  This is a minimal single-object Layer-2 application. In the common case (no other control
  effect) it reverts to owner; if the object is under a *second* still-active control effect it
  correctly keeps that one. The just-removed effect is already gone from `continuous_effects`
  before this runs, so it is excluded.

### Change 6: call site — fire promptly after control of the source changes

Control of a source changes only through effect resolution (another `GainControl`/steal
resolving on it) or the source leaving; **every** such path is immediately followed by
`sba::check_and_apply_sbas`. Site the expiry there, once, so it runs after every resolution at
all ~13 call sites:

- [x] **File**: `crates/engine/src/rules/sba.rs`, `check_and_apply_sbas` (L66-83). Call
  `crate::rules::layers::expire_while_you_control_source_effects(state)` **once at the top,
  before the `loop`** (before L68). Rationale: control never changes as a *result* of an SBA
  (no SBA changes control), so a single pre-loop call catches the post-resolution state; a
  reverted control feeds the SBA fixpoint that follows (an aura now on an illegal object, a
  soulbond needing re-check — both already SBA-handled via `obj.controller`). This is the
  same "housekeeping alongside the SBA cycle" placement the engine uses for commander-zone
  return and dungeon completion.
  - If `expire_while_you_control_source_effects` returns control-change events, `extend`
    `all_events` with them and run `abilities::check_triggers` on them (so "whenever you lose
    control"-style triggers, if any exist, fire). If no such `GameEvent` variant exists (grep
    found **none** — no `ControlChanged`/`GainedControl` variant), the pass returns `()` and no
    trigger wiring is needed; note this in the pass's doc comment.
  - **Do NOT** add this inside `apply_sbas_once`'s returned-events path — it must not gate the
    SBA fixpoint's termination (a no-op expiry every pass must leave `events` empty so the loop
    breaks).

## Version-Bump Checklist (machine-forced — read exact values from the failing gates)

Adding an `EffectDuration` variant moves **both** wire fingerprints: `EffectDuration` is in the

## Risks & Edge Cases

- **Never-resumes is the whole point.** The one-shot `retain`-out in
  `expire_while_you_control_source_effects` + `is_effect_active` returning `true` (not a live
  control check) together guarantee CR 611.2c. If a reviewer "simplifies" the arm to a live
  control check, the effect will incorrectly resume — the `does_not_resume` test guards this.
- **Phased-out source (CR 702.26e).** The ended-condition must NOT include `is_phased_in()`
  (unlike `WhileSourceOnBattlefield`). Getting this wrong permanently loses a stolen creature
  when the borrower's team phases out (Teferi's Protection). Guarded by
  `test_...survives_source_phase_out`.
- **Object identity (CR 400.7).** When the source leaves, it is a *new* object id in the new
  zone; the effect's `source` id no longer resolves → `unwrap_or(true)` = ended → revert. Do not
  "helpfully" try to follow the source across zones.
- **Stacked control effects.** `recompute_object_controller` reapplies remaining active
  `SetController` effects in timestamp order rather than blindly reverting to owner — otherwise a
  creature under two overlapping steals would wrongly snap to its owner when the first ends.
- **Call-site timing.** The expiry sits in `check_and_apply_sbas` (pre-loop). This is correct
  because control only changes via resolution, and every resolution calls `check_and_apply_sbas`
  afterward. If a future feature changes control **outside** a resolution/SBA boundary, this
  pass would lag — note this assumption in the function doc.
- **Pre-existing latent gap — file OOS-EF9-1 (do NOT fix in this PB):** `Effect::GainControl`
  with `WhileSourceOnBattlefield` (before this flip) and with `UntilEndOfTurn`
  (`sarkhan_vol.rs`, `zealous_conscripts.rs`, `karrthus_tyrant_of_jund.rs`) **never revert
  `obj.controller`** — the effect is removed but control persists. This is legal-but-wrong for
  any of those shipped `Complete`. The `recompute_object_controller` helper built here is
  exactly what would fix it (wire it into `expire_end_of_turn_effects` for removed
  `SetController` effects), but that changes existing Threaten-style behavior and existing
  golden scripts/tests — out of scope for this micro-PB. File OOS-EF9-1 with the roster
  (sarkhan_vol, zealous_conscripts, karrthus_tyrant_of_jund; plus a note that
  `test_gain_control_until_eot_expires` in primitive_pb32.rs is vacuous — it asserts effect
  removal but not control reversion) so a follow-up PB closes it with the helper already in place.
- **Yield honesty.** Confident Complete = olivia + dragonlord_silumgar (2). roil is an
  "attempt" gated on the optional-trigger wrapper; kellogg stays partial by design. Report the
