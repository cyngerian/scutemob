# M9.4 Fix Session Plan

Created: 2026-02-23
Source: `docs/mtg-engine-milestone-reviews.md` M9.4 review findings

## Session 1 -- Cascade and storm as triggered abilities (MR-M9.4-02, MR-M9.4-03, MR-M9.4-01)

This is the largest refactor: moving cascade and storm from inline resolution to
triggered abilities that go on the stack. Grouped together because they share the
same pattern (keyword -> triggered ability -> stack -> resolution).

- [x] MR-M9.4-02 (HIGH) -- Cascade resolved inline during CastSpell, not as triggered ability (CR 702.85a)
- [x] MR-M9.4-03 (HIGH) -- Storm resolved inline during CastSpell, not as triggered ability (CR 702.40a)
- [x] MR-M9.4-01 (HIGH) -- Cascade bottom-of-library move silently drops errors (`let _ =` in copy.rs:368)

Notes:
- Move cascade/storm from `casting.rs` to triggered ability path via `pending_triggers`
- Create new `StackObjectKind` variants or use existing `TriggeredAbility` for storm/cascade triggers
- Resolution of cascade trigger calls `copy::resolve_cascade`; resolution of storm trigger calls `copy::create_storm_copies`
- Fix the `let _ =` error suppression while refactoring cascade resolution
- Update cascade and storm tests to verify stack interaction (trigger on stack, priority before resolution)

Implementation notes (session 1):
- Added `StackObjectKind::CascadeTrigger { source_object, spell_mana_value }` and `StackObjectKind::StormTrigger { source_object, original_stack_id, storm_count }` to `state/stack.rs`
- Both variants push directly onto `stack_objects` in `casting.rs` (not via `pending_triggers`) since cascade/storm trigger at cast time
- `resolution.rs` handles both variants: `CascadeTrigger` calls `resolve_cascade`, `StormTrigger` calls `create_storm_copies`
- `let _ =` replaced with `if let Err(_) = ...` that acknowledges the error without silent data loss
- Hash coverage added in `state/hash.rs` for both new variants
- All 7 affected tests updated (3 cascade, 4 storm) to reflect trigger-on-stack behavior

## Session 2 -- Goad, cascade raw characteristics, cascade bottom-of-library (MR-M9.4-05, MR-M9.4-04, MR-M9.4-08)

Grouped by subsystem: cascade correctness fixes + goad implementation.

- [x] MR-M9.4-05 (MEDIUM) -- Effect::Goad only emits event, no continuous effect or combat enforcement
- [x] MR-M9.4-04 (MEDIUM) -- Cascade uses raw characteristics for card type and mana value check
- [x] MR-M9.4-08 (MEDIUM) -- Cascade "bottom of library" placement puts cards on top instead of bottom

Notes:
- Goad: add `goaded_by: Vector<PlayerId>` to `GameObject` or create continuous effect with `UntilNextTurnOf` duration; enforce in `combat.rs::handle_declare_attackers`; add hash coverage
- Cascade raw chars: replace `obj.characteristics.card_types` with `calculate_characteristics()` fallback
- Cascade bottom: add `Zone::push_to_bottom(id)` or equivalent zone positioning API

Implementation notes (session 2):
- Added `goaded_by: Vector<PlayerId>` field to `GameObject` in `game_object.rs` with `#[serde(default)]`
- `Effect::Goad` in `effects/mod.rs` now pushes the goading player into `obj.goaded_by` (dedup guard) in addition to emitting the `Goaded` event
- Goad expiry added to `untap_active_player_permanents` in `turn_actions.rs`: clears `goaded_by` at the start of the creature controller's turn (CR 701.15a)
- Combat enforcement in `combat.rs::handle_declare_attackers`:
  - Goaded creatures that can attack must be in the attackers list (CR 701.15b "must attack if able")
  - Goaded creatures attacking a goading player are rejected if a non-goading opponent target exists (CR 701.15b)
- `move_object_to_zone` updated to initialize `goaded_by: im::Vector::new()` (goad not preserved across zone changes, CR 400.7)
- New `move_object_to_bottom_of_zone` method on `GameState` uses `Zone::push_front` (position 0 = bottom)
- New `Zone::push_front` method in `zone.rs` inserts at position 0 of `Vector<ObjectId>`
- Cascade in `copy.rs` now calls `move_object_to_bottom_of_zone` for exiled cards not cast
- Cascade characteristic check now uses `calculate_characteristics(state, exile_id)` with raw fallback
- Hash coverage: `goaded_by.hash_into(hasher)` added to `impl HashInto for GameObject` in `hash.rs`
- Updated `GameObject` constructions in `builder.rs`, `effects/mod.rs` (token creation), and test files (`snapshot_perf.rs`, `zone_integrity.rs`, `commander_damage.rs`)
- 499 tests all pass; zero clippy warnings; formatting clean

## Session 3 -- Trigger doubler precision, overflow fix (MR-M9.4-07, MR-M9.4-06)

- [x] MR-M9.4-07 (MEDIUM) -- TriggerDoubler filter does not verify entering object type
- [x] MR-M9.4-06 (MEDIUM) -- `spells_cast_this_turn += 1` overflow in cascade

Notes:
- Add `entering_object_id: Option<ObjectId>` to `PendingTrigger`, check entering object types in `doubler_applies_to_trigger`
- Replace `+= 1` with `saturating_add(1)` in cascade resolution (copy.rs:326)

Implementation notes (session 3):
- Added `entering_object_id: Option<ObjectId>` field to `PendingTrigger` in `state/stubs.rs` with `#[serde(default)]`
- Added `entering_object_id.hash_into(hasher)` to `impl HashInto for PendingTrigger` in `state/hash.rs`
- Added `entering_object: Option<ObjectId>` parameter to `collect_triggers_for_event` in `rules/abilities.rs`; stored in each `PendingTrigger` created
- `PermanentEnteredBattlefield` handler passes `Some(*object_id)` as `entering_object` for both `SelfEntersBattlefield` and `AnyPermanentEntersBattlefield` trigger collection
- All other `collect_triggers_for_event` call sites pass `None` (no entering object concept)
- `doubler_applies_to_trigger` for `ArtifactOrCreatureETB` now requires `entering_object_id.is_some()`, uses `calculate_characteristics` with raw fallback to check the entering object has `CardType::Artifact` or `CardType::Creature` (CR 603.2d); removed the simplified/incorrect check
- `copy.rs:343`: replaced `ps.spells_cast_this_turn += 1` with `ps.spells_cast_this_turn = ps.spells_cast_this_turn.saturating_add(1)` (MR-M9.4-06)
- All 499 tests pass; zero clippy warnings; formatting clean
