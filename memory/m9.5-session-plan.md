## What M9.5 Delivers

- Standalone developer tool at `tools/replay-viewer/` -- Rust axum HTTP server + Svelte 5 frontend
- Replay engine that loads a game script JSON, pre-computes all steps via `process_command()`, stores `Vec<StepSnapshot>` in memory (cheap via im-rs structural sharing)
- REST API endpoints: `GET /api/scripts`, `GET /api/session`, `GET /api/step/:n`, `GET /api/step/:n/state`, `POST /api/load`
- View model serialization: `GameState` -> UI-friendly JSON (zones, players, turn info, combat)
- Svelte 5 SPA with step controls, keyboard navigation, rich zone visualization, event timeline, script browser
- Svelte components designed as props-based for reuse in M11 Tauri app
- No engine crate changes required -- this is a pure consumer of the engine's public API

### Data Model

Key new types (Rust backend):

    /// A complete replay session for one game script.
    /// Stored in axum's shared state behind Arc<RwLock<...>>.
    pub struct ReplaySession {
        /// The parsed game script.
        pub script: GameScript,
        /// Player name -> PlayerId mapping (from build_initial_state).
        pub player_map: HashMap<String, PlayerId>,
        /// Reverse map: PlayerId -> player name (for view model serialization).
        pub player_names: HashMap<PlayerId, String>,
        /// Pre-computed step snapshots (index 0 = initial state before any commands).
        pub steps: Vec<StepSnapshot>,
    }

    /// One step = one process_command() call + results.
    /// Stored cheaply via im-rs structural sharing on GameState.
    pub struct StepSnapshot {
        pub index: usize,
        /// The script action that produced this step (serialized for display).
        pub script_action: ScriptAction,
        /// The engine command that was sent (None for non-command actions like AssertState).
        pub command: Option<Command>,
        /// All events emitted by this command.
        pub events: Vec<GameEvent>,
        /// Game state after this command resolved.
        pub state_after: GameState,
        /// Assertion results if this step had an assert_state action.
        pub assertions: Option<Vec<AssertionResult>>,
    }

    /// Result of checking one assertion from an assert_state action.
    pub struct AssertionResult {
        pub path: String,
        pub expected: serde_json::Value,
        pub actual: serde_json::Value,
        pub passed: bool,
    }

### Interception Sites

**None in the engine crate.** This milestone is a pure consumer of the engine's public API. The replay viewer:
1. Reuses the `GameScript` schema from `mtg_engine::testing::script_schema`
2. Reuses `all_cards()`, `CardRegistry`, `GameStateBuilder`, `process_command()` from the engine's public API
3. Replicates the `build_initial_state` pattern from `crates/engine/tests/script_replay.rs` (the test harness function is in a `tests/` file, not a public module, so the replay viewer must have its own copy or the function must be moved to a public module)

**Key decision: `build_initial_state` extraction.** The function in `crates/engine/tests/script_replay.rs` is the exact logic the replay viewer needs. Two options:
1. **Copy it** into `tools/replay-viewer/src/replay.rs` (simple, no engine changes, but duplication)
2. **Move it** to `crates/engine/src/testing/` as a public function (DRY, but changes engine crate)

**Recommendation: Option 2 -- move to `crates/engine/src/testing/replay_harness.rs`** as a public function. This is a minor, non-breaking engine change that prevents duplication. The test in `crates/engine/tests/script_replay.rs` would then import from `mtg_engine::testing::replay_harness` instead. This also enables future tools to reuse the same function. Session 1 handles this extraction.

## Key Design Decisions

### View Model as Decoupling Layer

The frontend never sees raw `GameState`. The `view_model.rs` module converts `GameState` into a UI-friendly JSON shape. This means:
- Engine internal type changes do not break the frontend
- The same view model shape is reused in M11 Tauri app
- View model can include computed fields (e.g., "is this player active?") that the raw state does not

### Pre-computed Snapshots

All snapshots are computed at script load time. Stepping is a pure data lookup with zero latency. This is possible because:
- Game scripts are small (7-200 steps)
- im-rs structural sharing means storing all snapshots costs O(changed nodes), not O(full state x N)
- A 100-step script with 200 objects uses approximately 2MB total

### Props-Based Components

All Svelte components receive data via props and never fetch data internally. The data-fetching layer (`api.js` + `stores.js`) is the only part that differs between the stepper and the Tauri app. At M11, the Tauri app imports these same components and provides a different `api.js` that uses Tauri's `invoke()` instead of HTTP fetch.

### build_initial_state Extraction

Moving `build_initial_state()` and its helpers from `crates/engine/tests/script_replay.rs` to `crates/engine/src/testing/replay_harness.rs` prevents code duplication between the test harness and the replay viewer. Both will import from the same source. The test file retains `replay_script()` and `check_assertions()` since those are test-specific.

## Open Questions (for implementer to decide)

1. **CSS framework**: Plain CSS is recommended for a dev tool. Tailwind adds a build step and npm dependency complexity. If the implementer prefers Tailwind, install it during Session 2 step 4.

2. **Auto-play speed**: Session 2 mentions Space for auto-play at 1 step/second. The implementer can adjust the interval or add a speed control slider.

3. **Hidden information toggle**: Not required for M9.5. The viewer shows all information since it is a dev tool. A `?show_hidden=false` query parameter could be added in a future iteration to test the player-facing view.

4. **Vite dev proxy vs. static build**: During development, use Vite's dev server with proxy (`/api -> localhost:3030`). For production/CI, build the frontend to `dist/` and serve statically from axum. Both workflows should be documented in a brief README in `tools/replay-viewer/`.
